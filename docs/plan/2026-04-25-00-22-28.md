# Batch Child Scope 时间倒序稳定展示

## 摘要

将全局 queue 排序和 batch folder 内部 child scope 展示排序分离。queue 继续表达真实执行和创建顺序；batch folder children 按时间倒序展示，新月份/新日期在上，旧月份/旧日期在下。child job re-run 后仍停留在该 scope 的固定位置，不因为 `created_at` 变新而跳位。

## 关键改动

- 后端保留 existing planned/execution order，用于 batch child 创建和实际执行，不改变 job 调度语义。
- 后端 batch child display order 改为独立规则：planned child 优先，按 scope window `start DESC, end DESC` 排序。
- 同一个时间窗口内，用 planned scope index 作为稳定 tie-breaker，保留多 category 场景下的确定性顺序。
- 同一个 scope 的多个 attempt 在 `view=all` 下保持相邻，并按 `created_at DESC, id` 展示最新 attempt 在前。
- `list_child_jobs` 和 `list_jobs_read(parent_id=batch_id)` 使用新的 display order。
- 覆盖 `sync_papers_batch`、`find_repos_batch`、`refresh_metadata_batch`。
- 前端 batch folder children 不再按 `created_at` 二次排序。
- 不新增公开 `JobRead` 字段，不改数据库，不改 job 调度逻辑。

## 兜底逻辑

- 如果父 batch 无法解析，或者 child scope 不在 planned scopes 内，回退到 scope window 时间倒序，再按 `created_at` 和 `id` 排序。
- 非 batch child jobs、root jobs、queue jobs、普通 history rows、direct jobs、export jobs 保持现有排序行为。
- batch root job 的执行、rerun、child 创建顺序不改变。

## 测试

- 后端测试覆盖 `sync_papers_batch`、`find_repos_batch`、`refresh_metadata_batch` 都返回新月份在上、旧月份在下。
- 后端测试覆盖同一 scope 的 repair attempt 仍留在原 scope 位置，且 `view=all` 中同 scope attempts 相邻、最新 attempt 在前。
- 前端验证 child re-run 后在 batch folder list 原位置更新状态。
- 回归验证 queue 排序不受 scope 排序影响。

## 假设

- 本次不修改 `docs/design`。
- 本次只影响 batch child scope 展示排序，不改变 job 执行、重试或调度语义。
