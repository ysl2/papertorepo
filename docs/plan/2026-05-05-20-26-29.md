## SQL wildcard execution and status chip

### Requirement

When users run raw PostgreSQL SQL through the SQL search endpoint, normal SQL percent characters such as `ILIKE '%point cloud%'` must execute as SQL wildcards instead of being parsed by psycopg as client-side placeholders.

The SQL toolbar feedback chip should show only the overall execution state icon. Full non-table SQL feedback or error details remain available on hover, without expanding the toolbar or duplicating long text inline.

### Plan

1. Add backend coverage for PostgreSQL driver execution so raw SQL containing `%...%` is executed without psycopg placeholder parsing.
   - Verify: targeted SQL endpoint tests pass.
2. Update the PostgreSQL SQLAlchemy `exec_driver_sql` execution path to use no-parameter DBAPI execution for user SQL. Do not rewrite `%` in the SQL text and do not change the read-write raw cursor path, because both should preserve PostgreSQL SQL semantics.
   - Verify: targeted tests pass.
3. Keep SQL execution semantics unchanged: execute the whole SQL text as one unit, return only the final result/message, and rollback the whole unit on failure.
   - Verify: existing SQL endpoint tests continue to pass.
4. Replace the SQL feedback chip text with a status icon for success, failure, or cancelled, with the full message available through hover.
   - Verify: frontend lint/build check if available.
5. Move the chip to the right of the search field and loading/progress status, before the `SQL · N rows` summary.
   - Verify: frontend lint/build check if available.

### Out of scope

- Do not support psycopg `%s` parameter placeholder syntax in the user-facing SQL input.
- Do not add per-statement reporting, batch-runner behavior, savepoints, or partial-success state.
