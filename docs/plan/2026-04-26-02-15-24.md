# Drop source_github_url Plan

## Summary

Remove `source_github_url` because GitHub REST exposes `source`, but GitHub GraphQL `Repository` does not. After this change, refresh-metadata should return to the expected GraphQL-first path where successful GraphQL batches do not call REST.

## Key Changes

- Add Alembic migration `0015_drop_source_github_url` to drop `github_repos.source_github_url` from existing databases.
- Remove `source_github_url` from SQLAlchemy model, Pydantic `RepoRead`, frontend `Repo` type, GraphQL query, GraphQL/REST payload normalization, and metadata upsert field list.
- Keep `parent_github_url`; it is available from both REST and GraphQL for fork repositories.

## Test Plan

- Add migration test proving `0015` drops `source_github_url` and leaves `parent_github_url`.
- Update refresh-metadata GraphQL test so the query must not contain `source { url }`, GraphQL success has `graphql_fallbacks=0`, and REST is not called.
- Update public API/frontend type expectations so `source_github_url` is absent.
- Run `uv run python -m pytest`, `npm run build`, and `git diff --check`.

## Assumptions

- The stopped refresh-metadata job will not be resumed; a new job should be launched after this fix.
- Merge policy fields remain out of scope even if both REST and GraphQL expose them.
