# Stable SQL detail drawer loading

## Summary

- SQL row clicks still only identify the detail kind and id.
- Paper, job, and export details load through canonical APIs.
- Loading states use the normal drawer structure instead of a small `EmptyState`, so the drawer size stays stable while details load.
- Add a single export detail endpoint for cross-tab SQL export details.

## Key Changes

- Backend:
  - Add `GET /api/v1/exports/{export_id}` returning `ExportRead`.
  - Keep `/api/v1/exports/{export_id}/download` unchanged.
- Frontend:
  - Add selected export detail state and loading state.
  - Fetch `/api/v1/exports/:id` when an export is selected.
  - Replace paper/job/export loading `EmptyState` with drawer-shaped loading content.
  - Clear export detail/loading state when the drawer is closed or selection is reset.

## Test Plan

- Backend:
  - Add API tests for export detail success and 404.
  - Run public API tests.
- Frontend:
  - Run `npm run build`.
  - Manually verify cross-tab SQL clicks open a normal-sized drawer immediately, then fill details in place.

## Assumptions

- Export detail fields are the existing `ExportRead` fields.
- SQL rows should not be used to synthesize partial details.
