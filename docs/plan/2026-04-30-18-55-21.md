# SQL result detail drawer fix

## Context

SQL result row clicks are now resolved by PostgreSQL column provenance. A row can select a paper, job, or export even when the active tab is not the same table. The current drawer rendering and some detail-loading effects still depend on `previewTab`, so a SQL result that selects a job or export from the Papers tab can open the drawer with the paper empty state.

## Plan

1. Derive the drawer content from the selected entity state (`selectedPaperId`, `selectedJobId`, `selectedExportId`) instead of `previewTab`.
   - Verify: SQL rows sourced from `jobs.id` render the job detail drawer from any tab.
   - Verify: SQL rows sourced from `exports.id` render the export detail drawer from any tab.

2. Let selected job/paper detail loading follow selected entity state instead of active tab.
   - Verify: a SQL-selected job can hydrate full job details when the active tab is Papers or Exports.
   - Verify: normal tab-based row selection still works because only one selected entity is active.

3. Keep the change surgical.
   - Do not add column-name fallback.
   - Do not change database schema.
   - Do not add new detail pages for unsupported tables.

4. Run focused verification.
   - Frontend build.
   - Existing SQL endpoint tests.
   - Full backend test suite if feasible.
   - Diff whitespace check.
