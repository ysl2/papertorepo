# SQL 结果详情页改为基于数据库列来源判断

## Summary

修正 SQL 模式下详情页打开逻辑：不再按当前 tab 或列名猜测实体类型，而是由后端在 `/api/v1/sql` 响应中返回 PostgreSQL 列来源 metadata。前端只根据真实来源映射到已有的 Paper / Job / Export 详情页；其他来源或无来源的结果不打开详情。

## Key Changes

- 后端 `SqlResponse` 增加 `column_sources`，与 `columns` 一一对应，包含 `source_schema`、`source_table`、`source_column`。
- PostgreSQL 路径用 `cursor.pgresult.ftable(index)` / `ftablecol(index)` 读取来源表 OID 和列号，再查 `pg_catalog` 映射为 schema/table/column。
- 保留现有展示列名和重复列名去重逻辑；`column_sources[index]` 始终对应 `columns[index]`。
- 前端 `SqlResponse` / `SqlResult` 类型增加 `columnSources`。
- SQL 行点击按结果列顺序扫描来源，仅支持：
  - `papers.arxiv_id` -> Paper drawer
  - `jobs.id` -> Job drawer
  - `exports.id` -> Export drawer
- 不做列名 fallback；无 provenance、表达式、聚合、literal、其他表都不打开详情。
- SQL 模式点击当前已打开的同一个实体时关闭详情页。

## Test Plan

- SQLite SQL 端点测试断言 SELECT 响应包含 `column_sources`，来源为 `null`。
- 增加 provenance helper 单元测试，覆盖 `papers.arxiv_id`、`jobs.id`、`exports.id`、重复展示列名和无来源列。
- 保持现有 SELECT、UPDATE、非法 SQL、非 JSON 类型序列化测试通过。
- `npm run build` 通过。
- 手工实测 Docker/PostgreSQL：任意 tab 查询 papers/jobs/exports 普通列可打开对应详情，聚合查询不打开详情，同一实体二次点击关闭详情。

## Assumptions

- 当前生产/Docker 环境是 PostgreSQL；SQLite 只作为测试和轻量本地兼容路径。
- 暂时只支持已有三种详情页，不为其他表新增详情页。
- 不改数据库 schema，不要求用户写 `AS`，也不基于列名猜测实体归属。
