# SQL aggregate summary provenance fix

## Summary

- Fix SQL summary source-table detection so aggregate or expression result columns do not look like row-level table results.
- Keep `x / total rows` only when every result column is directly sourced from one supported table: `papers`, `jobs`, or `exports`.
- Keep `SQL · x rows` for unknown, mixed, aggregate, expression, constant, or unsupported table provenance.

## Key Changes

- Update the frontend `uniqueSupportedSqlSourceTable()` helper only.
- Return `null` if any column lacks a supported `source_table`.
- Return the supported table only when all columns agree on the same source table.
- Do not change backend provenance, SQL cancellation/loading, or detail drawer behavior.

## Test Plan

- Run `npm run build`.
- Run backend tests affected by prior dashboard/API changes if needed.
- Manually verify:
  - Plain `papers/jobs/exports` selects still show `x / total rows`.
  - `SELECT primary_category, COUNT(*) AS paper_count FROM papers GROUP BY primary_category ORDER BY paper_count DESC;` shows `SQL · x rows`.

## Assumptions

- Backend column provenance is correct; the bug is only frontend interpretation.
- `x / total rows` should mean every visible SQL result column is directly sourced from one supported base table.
