# 拆分 Frontend Runtime Config 端点

## Summary

本次只做语义拆分：新增 `GET /api/v1/runtime-config`，把当前 `/health` 中属于前端启动配置的字段迁过去，前端行为保持不变。

## Key Changes

- 后端新增 `RuntimeConfigRead` schema 和 `GET /api/v1/runtime-config`。
- `runtime-config` 返回 `default_categories`、`sql_search_mode`、`step_providers`。
- `/health` 保留 `app_name`、`api_prefix`、`database_dialect`、`queue_mode`、`github_auth_configured`、`effective_github_min_interval_seconds`。
- 前端新增 `RuntimeConfig` 类型和 state；页面启动时读取 `/health` 和 `/runtime-config`。
- 前端原来依赖 `health.default_categories`、`health.sql_search_mode`、`health.step_providers` 的地方改为依赖 `runtimeConfig`。
- 不做 `.env.example` 常量整理、不做 `VITE_*`、不改 SQL 权限逻辑、不改 Docker/env 读取方式。

## Test Plan

- 修改后端 public API 测试：
  - `/api/v1/health` 仍返回健康/运行状态字段。
  - `/api/v1/runtime-config` 返回 `default_categories`、`sql_search_mode`、`step_providers`。
  - `SQL_SEARCH_MODE=read_only` 时 `runtime-config.sql_search_mode` 为 `read_only`。
- 前端构建验证：`npm --prefix frontend run build`。
- 后端测试验证：`cd backend && uv run python -m pytest`。

## Assumptions

- 这次允许 `/health` 响应字段发生语义清理，不继续保留 `default_categories`、`sql_search_mode`、`step_providers` 的兼容副本。
- 如果 `/runtime-config` 请求失败，前端按现有健康信息加载失败的方式显示错误，不静默降级。
- `runtime-config` 是公开给浏览器的配置端点，后续新增字段必须默认按“可公开信息”处理。
