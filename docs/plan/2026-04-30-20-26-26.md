# SQL query loading and cancellation

## Summary

- Add a single-request SQL controller in the frontend: one SQL query can be `running` or `canceling` at a time.
- Show clear SQL execution feedback: disable the search field, show `Cancel` / `Canceling...` inside the search box, and overlay an indeterminate progress bar at the top of the grid body.
- Add best-effort PostgreSQL cancellation on the backend so `Cancel` means more than stopping the browser wait.
- Do not add pagination, streaming, real percentage progress, query history, queues, or schema changes.

## Key Changes

- Extend SQL API:
  - `SqlRequest` gets optional `request_id`.
  - Add `POST /api/v1/sql/{request_id}/cancel`.
  - Add `SqlCancelResponse` with `ok`, `request_id`, `cancel_requested`, and `message`.
- Backend:
  - Register running SQL requests by `request_id` in a thread-safe in-memory registry.
  - Reject duplicate active `request_id` without executing another SQL statement.
  - For PostgreSQL, cancel endpoint calls psycopg `cancel_safe(timeout=2.0)` when available.
  - Clean up registry entries in `finally`.
  - Keep SQLite behavior simple; no real SQLite query cancellation.
- Frontend:
  - Track `idle | running | canceling`, active `request_id`, and active `AbortController`.
  - Enter only submits when idle.
  - Running/canceling disables input and prevents a second SQL query.
  - Cancel calls the backend cancel endpoint and aborts the in-flight fetch.
  - Ignore stale SQL responses whose `request_id` no longer matches.
  - Reset and tab changes cancel active SQL before clearing the SQL view.

## Test Plan

- Backend tests:
  - SQL request with `request_id` registers and cleans up.
  - Duplicate active `request_id` returns `ok=false`.
  - Cancel endpoint calls `cancel_safe(timeout=2.0)` for a running PostgreSQL request.
  - Cancel endpoint for an unknown request returns benign success with `cancel_requested=false`.
  - Existing SQL endpoint tests still pass.
- Frontend/build verification:
  - `npm run build`.
  - Slow SQL shows disabled search, `Cancel`, and grid-top indeterminate progress.
  - Cancel transitions through `Canceling...`, preserves the SQL text, and allows a later rerun.
  - Stale canceled responses do not update the table.
  - Reset/tab changes cancel active SQL and return UI to an operable state.

## Assumptions

- Production uses PostgreSQL; SQLite is mainly for tests and local lightweight runs.
- PostgreSQL cancellation is best-effort. A successful client cancel request does not guarantee the server canceled instantly, but it is the correct minimal implementation for this feature.
